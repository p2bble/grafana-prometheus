## 1. 배경 및 목표

KVM 하이퍼바이저 3대에 분산된 전체 VM(총 16개)에 대한 무중단 자동 백업 체계를 구축한다.

- **백업 방식:** 가동 중 VM은 external snapshot → backing file 복사 → blockcommit 병합 (무중단), 정지 VM은 qemu-img 직접 복사
- **백업 저장소:** NAS NFS 마운트 `/storage/vm_backup/` (42T, 서버 공유)
- **세대 관리:** 최신 3세대 유지, 이전 세대 자동 삭제
- **스케줄:** pdkcld1 cron `0 1 * * 0` (매주 일요일 01:00)

## 2. 첫 자동 실행 (2026-05-03 01:00) — 발견된 문제 및 해결

### 2-1. 순차 실행 시 소요 시간 초과

**문제:** vm-backup-all.sh가 순차 실행 방식으로 VM 12개를 처리할 경우 예상 소요 시간이 4일 이상

**해결:** MAX_PARALLEL=3 병렬 실행 방식으로 전환

```bash
# wait_slot(): 실행 중 PID 수가 MAX_PARALLEL 이상이면 60초 대기 후 재확인
wait_slot() {
    while [ "${#PIDS[@]}" -ge "${MAX_PARALLEL}" ]; do
        # 완료된 PID 수확 후 슬롯 재계산
        ...
        sleep 60
    done
}
```

**결과:** 병렬 처리로 전체 소요 시간 약 1/3 단축

### 2-2. croms-rx-UXUI — volume-type disk 스냅샷 실패

**문제:** vdb(`croms-simulator-data`)가 libvirt storage pool volume 타입으로 등록되어 있어 snapshot 생성 시 오류 발생

```
error: unsupported configuration: cannot generate external snapshot name
for disk 'vdb' without source
```

**원인:** libvirt는 volume 타입 disk에 대해 overlay 파일명을 자동 생성하지 못함

**해결:** `--diskspec` 옵션으로 overlay 경로를 명시적으로 지정

```bash
DISKSPEC_ARGS=()
for i in "${!PRE_PATHS[@]}"; do
    DISKSPEC_ARGS+=(--diskspec "${PRE_TARGETS[$i]},file=${PRE_PATHS[$i]}.${SNAP_TAG}")
done
virsh snapshot-create-as "${VM_NAME}" "${SNAP_TAG}" \
    --disk-only --atomic --no-metadata \
    "${DISKSPEC_ARGS[@]}"
```

`resolve_vol_path()` 함수로 volume 이름 → 실제 파일 경로 변환 후 diskspec에 사용

### 2-3. croms-rx-core-dev — 이중 백업 충돌

**문제:** 병렬 전환 과도기에 순차 스크립트와 병렬 스크립트가 동시에 같은 VM 백업을 시도하면서 overlay chain이 2단계로 형성됨

```
original.qcow2 ← overlay1 (run1) ← overlay2 (run2)
```

**해결 과정:**
1. run2 프로세스 강제 종료
2. `virsh blockcommit vda --base overlay1 --active --pivot --wait` → overlay2 제거
3. `virsh blockcommit vdb --active --pivot --wait` → overlay chain 전체 original로 병합
4. overlay 파일 수동 삭제

**결과:** VM 디스크 체인 복구 완료 (데이터 손실 없음). vda 백업(34G)만 존재, vdb 재백업 필요

## 3. 스크립트 개선

### 3-1. mtime 기반 변경 감지 (불필요한 백업 스킵)

**배경:** 변경이 없는 VM도 매주 동일하게 복사하면 스토리지·IO 낭비

**방식:** 마지막 백업 디렉토리 생성 시각보다 qcow2 파일 mtime이 오래됐으면 스킵

```bash
needs_backup() {
    local last_dir
    last_dir=$(ls -dt "${BACKUP_BASE}/${VM_NAME}"/[0-9]* 2>/dev/null | head -1)
    [ -z "${last_dir}" ] && return 0  # 첫 백업이면 무조건 진행

    collect_disks "${VM_NAME}"
    for path in "${DISK_PATHS[@]}"; do
        [ "${path}" -nt "${last_dir}" ] && return 0  # 변경 감지
    done

    log "=== 변경 없음 — 스킵 (마지막 백업: $(basename "${last_dir}")) ==="
    rm -rf "${BACKUP_DIR}"
    return 1
}

needs_backup || exit 0
```

**적용 범위:** pdkcld1, vm-server-01, gpu-server 전체 배포

> **참고:** running VM은 QEMU flush 시 mtime이 갱신될 수 있어 false positive 가능성 있음. 실제 IO가 없는 running VM에서는 드물게 발생.

## 6. 스크립트 배포 주의사항

- **CRLF 문제:** Windows에서 작성한 파일을 SCP로 배포할 경우 `\r` 포함. 배포 전 반드시 CRLF→LF 변환 필요
  ```bash
  # 서버에서 직접 수정
  sudo sed -i 's/\r//' /usr/local/bin/vm-live-backup.sh
  # 문법 확인
  sudo bash -n /usr/local/bin/vm-live-backup.sh
  ```
- **배포 대상:** pdkcld1, vm-server-01, gpu-server 3개 서버 동일 버전 유지
